# Registry Integration Guide

> **Version:** 1.0  
> **Status:** Planning Phase

This document describes the planned integration with OpenTofu Registry, Terraform Provider ecosystem, and the StackKits Registry.

## Overview

StackKits integrates with multiple registries to provide:

1. **OpenTofu/Terraform Providers** - Infrastructure provisioning
2. **StackKits Registry** - Community StackKits
3. **Service Alternatives** - Replaceable service definitions

## OpenTofu Registry Integration

### Provider Discovery

StackKits automatically resolve providers from the OpenTofu Registry:

```hcl
# Generated by StackKit
terraform {
  required_providers {
    # Docker container management
    docker = {
      source  = "kreuzwerker/docker"
      version = "~> 3.0"
    }
    
    # Remote execution
    null = {
      source  = "hashicorp/null"
      version = "~> 3.2"
    }
    
    # DNS management (optional)
    cloudflare = {
      source  = "cloudflare/cloudflare"
      version = "~> 4.0"
    }
    
    # Proxmox VMs (future)
    proxmox = {
      source  = "bpg/proxmox"
      version = "~> 0.40"
    }
  }
}
```

### Available Providers

| Provider | Purpose | Registry Source |
|----------|---------|-----------------|
| `docker` | Container management | `kreuzwerker/docker` |
| `null` | Remote execution | `hashicorp/null` |
| `local` | Local files | `hashicorp/local` |
| `random` | Random values | `hashicorp/random` |
| `cloudflare` | DNS/CDN | `cloudflare/cloudflare` |
| `proxmox` | VM management | `bpg/proxmox` |
| `hcloud` | Hetzner Cloud | `hetznercloud/hcloud` |

### Provider Pinning

```yaml
# kombination.yaml
providers:
  docker:
    version: "3.0.2"     # Exact version
  cloudflare:
    version: "~> 4.0"    # Compatible versions
  proxmox:
    version: ">= 0.40"   # Minimum version
```

## StackKits Registry

### Registry API

The StackKits Registry provides a REST API for StackKit discovery:

```
Base URL: https://registry.stackkits.dev/v1/

Endpoints:
  GET  /stackkits                    # List all StackKits
  GET  /stackkits/{name}             # Get StackKit details
  GET  /stackkits/{name}/versions    # List versions
  GET  /stackkits/{name}/{version}   # Get specific version
  POST /stackkits                    # Publish (authenticated)
```

### Publishing StackKits

```bash
# Validate StackKit
stackkit validate ./my-stackkit

# Publish to registry (requires authentication)
stackkit publish ./my-stackkit

# Publishing steps:
#   1. Validate structure and schemas
#   2. Run automated tests
#   3. Sign package with GPG key
#   4. Upload to registry
#   5. Generate checksums
```

## Service Alternatives

### Concept

Many StackKits define "service categories" with replaceable implementations:

```cue
// Service category definition
#PhotoService: #Immich | #Ente | #PhotoPrism

// Each implementation follows the same interface
#Immich: base.#ServiceDefinition & {
    name:     "immich"
    type:     "photo-cloud"
    image:    "ghcr.io/immich-app/immich-server"
    ports:    [{host: 2283, container: 3001}]
    volumes:  [...]
    // ...
}

#Ente: base.#ServiceDefinition & {
    name:     "ente"
    type:     "photo-cloud"
    image:    "ghcr.io/ente-io/photos"
    ports:    [{host: 2283, container: 8080}]
    volumes:  [...]
    // ...
}
```

### Replacement Syntax

```bash
# Via CLI flag
stackkit apply --replace immich=ente

# Via configuration file
# kombination.yaml
services:
  photos:
    alternative: ente  # Use Ente instead of Immich
    config:
      # Ente-specific configuration
      encryption_key: "${SECRET_ENCRYPTION_KEY}"
```

### Available Alternatives

| Category | Default | Alternatives |
|----------|---------|--------------|
| `photo-cloud` | Immich | Ente, PhotoPrism, LibrePhotos |
| `media-server` | Jellyfin | Plex, Emby |
| `paas-platform` | Dokploy | Coolify, Caprover |
| `container-ui` | Dockge | Portainer, Yacht |
| `monitoring` | Uptime Kuma | Beszel, Grafana |
| `dns-server` | AdGuard Home | Pi-hole, Blocky |
| `vpn` | WireGuard | Tailscale, Netbird |

## Provider Extensions

### Custom Providers

StackKits can define custom provider requirements:

```yaml
# stackkit.yaml
providers:
  required:
    - docker
    - null
  optional:
    - cloudflare:
        when: network.mode == "public"
    - proxmox:
        when: nodes[].type == "vm"
```

### Provider Configuration

```yaml
# kombination.yaml
providers:
  cloudflare:
    api_token: "${CLOUDFLARE_API_TOKEN}"
    zone_id: "${CLOUDFLARE_ZONE_ID}"
  
  proxmox:
    endpoint: "https://pve.local:8006"
    username: "root@pam"
    password: "${PROXMOX_PASSWORD}"
```

## Offline Mode

### Caching

```bash
# Download StackKit for offline use
stackkit cache add base-homelab

# Download all dependencies
stackkit cache sync

# Use cached StackKit
stackkit init base-homelab --offline

# Cache location
# Linux:   ~/.cache/stackkit/
# macOS:   ~/Library/Caches/stackkit/
# Windows: %LOCALAPPDATA%\stackkit\
```

### Air-gapped Deployments

```bash
# Create offline bundle
stackkit bundle create base-homelab --output bundle.tar.gz
# Bundle includes:
#   - StackKit definition
#   - Provider binaries
#   - Docker images (optional)

# Transfer to air-gapped system
scp bundle.tar.gz user@server:/tmp/

# Install from bundle
stackkit bundle install /tmp/bundle.tar.gz
stackkit apply --offline
```

## Security

### Signature Verification

All registry artifacts are signed:

```bash
# Verify StackKit signature
stackkit verify kombihq/base-homelab

# Import trusted keys
stackkit trust add kombihq

# Skip verification (not recommended)
stackkit init base-homelab --skip-verify
```

## Troubleshooting

### Registry Connection Issues

```bash
# Test connectivity
stackkit registry ping

# Use alternate registry
stackkit init base-homelab --registry https://mirror.example.com

# Debug mode
STACKKIT_DEBUG=1 stackkit init base-homelab
```

### Version Resolution

```bash
# List available versions
stackkit versions kombihq/base-homelab

# Force specific version
stackkit init kombihq/base-homelab@1.0.0 --force
```
