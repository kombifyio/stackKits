# StackKits Architecture

> **Version:** 2.0  
> **Last Updated:** 2026-01-10  
> **Status:** Production Ready

## Overview

StackKits is a declarative infrastructure blueprint system that combines **CUE** for validation and schema definition, **OpenTofu** for infrastructure provisioning, and optionally **Terramate** for orchestration. This architecture enables infrastructure-as-code (IaC) that is validated before execution, ensuring safe and predictable deployments.

## Core Components

### 1. CUE Schema Layer

CUE is a data validation language that unifies schema, configuration, and policy:

```cue
// Example: Service Definition Schema
#ServiceDefinition: {
    name:        =~"^[a-z][a-z0-9-]+$"  // DNS-compatible
    image:       string
    tag:         string | *"latest"     // default value
    ports:       [...#PortMapping]
    healthCheck: #HealthCheck
}
```

**Key Benefits:**
- **Type Safety:** Catch configuration errors before deployment
- **Cross-Field Validation:** Ensure `minimum < maximum` relationships
- **Default Values:** Smart defaults reduce boilerplate
- **Composability:** Schema inheritance and extension

### 2. OpenTofu Provider Layer

OpenTofu (Terraform-compatible) provides the actual infrastructure provisioning:

| Provider | Purpose | Usage |
|----------|---------|-------|
| `kreuzwerker/docker` | Docker containers/networks | Service deployment |
| `hashicorp/null` | Remote execution | OS bootstrap |
| `hashicorp/local` | Local file generation | Config files |
| `cloudflare/cloudflare` | DNS (optional) | Public domains |

### 3. Terramate Orchestration (Optional)

For multi-node deployments, Terramate provides:

- **Stack Management:** Organize multiple OpenTofu configurations
- **Drift Detection:** `terramate run --sync-drift-status`
- **Selective Updates:** Only deploy changed stacks
- **Parallel Execution:** Speed up multi-node deployments

## Prerequisites

StackKits requires only two runtime dependencies:

| Dependency | Version | Purpose |
|------------|---------|---------|
| **Docker** | 24.0+ | Container runtime |
| **OpenTofu** | 1.6+ | Infrastructure provisioning |

Optional:
- **Terramate** 0.6+ (for advanced/multi-node mode)
- **CUE** 0.9+ (for local schema validation/development)

## Layer Architecture

StackKits uses a three-layer architecture:

### Layer 1: Core (Shared)

The `base/` directory contains shared schemas and templates:

```
base/
├── stackkit.cue        # Base StackKit definition
├── system.cue          # System configuration (timezone, locale)
├── network.cue         # Network settings (DNS, NTP, VPN)
├── security.cue        # Security policies (SSH, firewall, TLS)
├── observability.cue   # Monitoring and logging
├── bootstrap/          # OS preparation templates
│   └── _bootstrap.tf.tmpl
├── network/            # Network mode templates
│   ├── _local.tf.tmpl  # IP-based, self-signed TLS
│   └── _public.tf.tmpl # Domain-based, ACME TLS
└── lifecycle/          # Day-2 operations
    └── _drift.terramate.hcl
```

### Layer 2: StackKits

Each StackKit extends the base with specific service definitions:

```
base-homelab/
├── stackkit.yaml       # Metadata and requirements
├── stackfile.cue       # Main schema (extends base)
├── services.cue        # Service definitions
├── defaults.cue        # StackKit-specific defaults
├── variants/
│   ├── os/             # OS-specific configurations
│   │   ├── ubuntu-24.cue
│   │   └── debian-12.cue
│   └── compute/        # Resource tier configurations
│       └── compute.cue
└── templates/
    ├── simple/         # Single-file OpenTofu config
    │   └── main.tf
    └── advanced/       # Terramate-orchestrated stacks
        └── stacks/
```

### Layer 3: Add-ons

Optional functionality that extends any StackKit:

```
addons/
├── monitoring/         # Prometheus, Grafana
├── vpn-overlay/        # Tailscale, WireGuard
└── backup-restic/      # Automated backups
```

## Data Flow

### 1. User Intent → Validated Config

```yaml
# User provides: kombination.yaml
name: my-homelab
stackkit: base-homelab
variant: default
domain: homelab.local
nodes:
  - name: server-1
    os: ubuntu-24
    ip: 192.168.1.100
```

### 2. CUE Validation

```bash
# Schema validation
cue vet ./base-homelab/... spec.yaml

# Export resolved configuration
cue export ./base-homelab/... -e resolvedSpec
```

The CUE engine:
1. Validates against `#BaseStackKit` schema
2. Applies OS variant (`#Ubuntu24Variant`)
3. Selects compute tier based on resources
4. Merges service definitions
5. Outputs complete, validated configuration

### 3. OpenTofu Generation

The validated configuration feeds into OpenTofu templates:

```hcl
# Generated: main.tf
resource "docker_container" "traefik" {
  name  = "traefik"
  image = docker_image.traefik.image_id
  
  ports {
    internal = 80
    external = 80
  }
  
  labels = {
    "managed-by" = "stackkit"
    "stackkit"   = "base-homelab"
  }
}
```

### 4. Execution

```bash
# Simple mode
cd templates/simple
tofu init
tofu plan -out=plan.tfplan
tofu apply plan.tfplan

# Advanced mode (multi-node)
terramate run -- tofu init
terramate run -- tofu apply -auto-approve
```

## IaC-First Principle

StackKits follows an **IaC-First** architecture:

| Aspect | IaC-First ✅ | Agent-First ❌ |
|--------|--------------|----------------|
| Execution | `tofu apply` | Shell commands |
| State | OpenTofu state file | Custom database |
| Drift Detection | `tofu plan -detailed-exitcode` | Custom polling |
| Rollback | `tofu destroy` or state restore | Re-run scripts |
| Providers | 3000+ Terraform providers | Custom implementations |
| Maintenance | Community-maintained | Self-maintained |

### Why IaC-First?

1. **Declarative:** Define what you want, not how to get there
2. **Idempotent:** Run the same config multiple times safely
3. **Auditable:** State file shows exactly what's deployed
4. **Recoverable:** Rollback is trivial with state history

## Network Modes

### Local Mode (IP-Based)

For homelabs without public domains:

- Services accessible via `http://<ip>:<port>`
- Self-signed TLS certificates (optional)
- No external DNS required

### Public Mode (Domain-Based)

For homelabs with public domains:

- Services accessible via `https://service.domain.com`
- Automatic ACME/Let's Encrypt certificates
- Requires port 80/443 accessible from internet

## Security Model

StackKits implements security-by-default:

```cue
security: {
    ssh: {
        permitRootLogin: "no"
        passwordAuth:    false
        pubkeyAuth:      true
        maxAuthTries:    3
    }
    firewall: {
        enabled:         true
        defaultInbound:  "deny"
        defaultOutbound: "allow"
    }
    container: {
        runAsNonRoot:    true
        privileged:      false
        noNewPrivileges: true
    }
}
```

## Integration with KombiStack

When used with KombiStack (the web UI):

1. **UI Wizard** generates `kombination.yaml`
2. **Unifier Engine** validates via CUE schemas
3. **Worker Agent** executes OpenTofu commands
4. **Dashboard** displays deployment status

Standalone mode (CLI) bypasses the web UI while using the same StackKit definitions.
